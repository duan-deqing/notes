# 异常处理

## 异常层次结构与常见异常类型

程序运行时难免遇到错误。Python 把一切错误都抽象成对象，称为 **异常**。所有内置异常都继承自 `BaseException`，而绝大多数程序错误的基类是 `Exception`。了解异常的分类，有助于你精确捕获、恰当处理。

### 异常继承结构（简版）

```text
BaseException
 ├── SystemExit
 ├── KeyboardInterrupt
 ├── GeneratorExit
 └── Exception
      ├── StopIteration
      ├── ArithmeticError
      │    ├── ZeroDivisionError
      │    └── OverflowError
      ├── LookupError
      │    ├── IndexError
      │    └── KeyError
      ├── TypeError
      ├── ValueError
      ├── NameError
      ├── IOError / OSError
      │    ├── FileNotFoundError
      │    └── PermissionError
      └── ... 
```

从图中可以看出，如果抓 `except ArithmeticError`，会同时捕获 `ZeroDivisionError` 和 `OverflowError`，因为它们都是该基类的子类。

### 最常见的异常类型

| 异常类              | 触发场景                                                     |
| :------------------ | :----------------------------------------------------------- |
| `SyntaxError`       | 代码语法错误，解释器无法解析。该异常在解析阶段抛出，通常无法用 `try` 捕获 |
| `NameError`         | 使用了未定义的变量名                                         |
| `TypeError`         | 对类型不兼容的对象进行运算或调用                             |
| `ValueError`        | 类型正确但值不合法（如 `int("abc")`）                        |
| `ZeroDivisionError` | 除法或取模时除数为零                                         |
| `IndexError`        | 序列索引超出范围                                             |
| `KeyError`          | 字典中查找不存在的键                                         |
| `FileNotFoundError` | 打开文件时路径不存在                                         |
| `AttributeError`    | 对象不存在所访问的属性                                       |

??? note "示例"
    ```python
    x = 10
    y = 0
    print(x / y)            # ZeroDivisionError

    data = {"name": "Ada"}
    print(data["age"])      # KeyError
    
    int("xyz")              # ValueError
    ```

理解异常层次不仅可以帮你写出更稳健的 `except`，还会在大型系统里让错误报告更有层次感。

## try-except 捕获与异常匹配

`try` 块用于包裹可能抛出异常的代码；`except` 块定义如何处理某类异常。语法如下：

```python
try:
    可能出错的代码
except 异常类 as 变量:
    处理代码
```

### 基本捕获

```python
try:
    num = int(input("请输入一个整数: "))
    print(f"输入了 {num}")
except ValueError:
    print("不是合法的整数")
```

`as` 可以捕获异常对象，查询详细信息：

```python
try:
    val = 1 / 0
except ZeroDivisionError as e:
    print(f"发生错误: {e}")  # division by zero
```

### 捕获多种异常

可以使用多个 `except` 分支，或在一个 `except` 后用元组列出多个异常类。

```python
try:
    a = int(input("a: "))
    b = int(input("b: "))
    print(a / b)
except ValueError:
    print("请输入数字")
except ZeroDivisionError:
    print("除数不能为零")
except (TypeError, KeyError):    # 合并捕获
    print("类型或键错误")
```

!!! tip "**匹配规则**"
    从上到下依次检查异常类型，先匹配到的分支被执行，后面的不再检查。因此，子类异常必须放在父类异常 **之前**。
    ```python
    try:
        ...
    except Exception:          # 父类
        print("捕获所有")       # 这会吞掉所有异常，后面的无法执行
    except ValueError:         # 永远不会执行
        print("不会到这里")
    ```

!!! warning "避免裸捕获"
    除非你清楚自己在做什么，否则 **不要使用空的 `except:` 或 `except Exception:` 并什么都不做**。它会吞噬所有异常（包括 `KeyboardInterrupt` 和 `SystemExit`），让程序难以调试及退出。
    ```python
    # 危险做法
    try:
        risky_op()
    except:
        pass          # 错误被隐藏，bug 难查
    ```
    如需捕获所有异常，至少记录日志并再次抛出或妥善关闭资源。

### 常见实践

捕获具体异常，恢复或提供友好信息；无法恢复时重新抛出或让异常向上传播。只捕获你预计会发生的异常，未知错误应“大声音”暴露

## else 与 finally资源清理语义

除了 `try` 和 `except`，Python 提供了 `else` 和 `finally` 子句，让错误处理逻辑更清晰。

### else：仅在无异常时执行

`else` 块中的代码只会在 `try` 块 **没有抛出异常** 时运行。将依赖成功操作的后续逻辑放在 `else` 里，可以更明确地表达“仅当一切正常时做这件事”。

```python
try:
    f = open("data.txt", "r")
except FileNotFoundError:
    print("文件不存在")
else:
    content = f.read()
    print(f"读取到 {len(content)} 字节")
    f.close()
```

这样做的好处是：把可能出错的 `open` 放在 `try` 里，而依赖 `f` 被正确初始化的操作放在 `else`，如果 `open` 失败就不会执行它们，避免 `NameError`。

### finally：无论是否异常都执行

`finally` 块的代码 **总是** 会被执行，即便 `try` 中有 `return`、`break` 或没有捕获的异常。它适合做清理工作，比如关闭文件、释放锁、断开网络连接。

```python
f = None
try:
    f = open("log.txt", "a")
    f.write("日志条目")
except OSError as e:
    print(f"写入失败: {e}")
finally:
    if f:
        f.close()
    print("清理完成")
```

!!! tip
    但是，更推荐的方式是使用 `with` 语句管理资源（见[第九章](./09-文件与数据持久化.md#)），它内部使用了 `finally` 机制保证释放。手动 `try...finally` 在涉及非文件资源（如锁）时仍然常用。

### 完整结构顺序

```python
try:
    ...
except SomeException:
    ...
else:
    ...
finally:
    ...
```

处理逻辑一目了然：`try` 里是可能出错的代码，`except` 处理错误，`else` 是无错误时的成功逻辑，`finally` 是最终的保洁工作。

## raise主动抛出异常

不仅解释器可以抛出异常，你也可以用 `raise` 主动抛出，让调用者知道某个错误条件已经发生。

### 基本语法

```python
raise SomeException("错误描述")
```

示例：验证函数参数

```python
def set_age(age):
    if not isinstance(age, int):
        raise TypeError("年龄必须是整数")
    if age < 0 or age > 150:
        raise ValueError(f"年龄值不合理: {age}")
    print(f"年龄设置为 {age}")

set_age(-5)   # ValueError: 年龄值不合理: -5
```

### 重新抛出异常

在 `except` 块内使用 `raise` 不加任何参数，会原样 **重新抛出当前异常**，保留原始堆栈追踪。这在需要记录日志后仍让上层处理错误时非常有用。

```python
try:
    do_something()
except ValueError as e:
    print(f"记录日志: {e}")
    raise          # 重新抛出同一个异常
```

!!! tip
    如果用 `raise e` 重新抛出，会丢失原始堆栈，难以定位问题根源，因此推荐裸 `raise`。

### 异常链：raise ... from

当一个异常由另一个异常引发时，可以用 `from` 将原异常设置为“原因”，方便调试。

```python
try:
    result = parse_data(input_string)
except ValueError as ve:
    raise RuntimeError("数据解析失败") from ve
```

查看堆栈时会清楚看到 `RuntimeError` 是由 `ValueError` 直接导致的。如果不想显式链，可使用 `from None` 抑制上下文显示。

## **\* 自定义异常类的设计**

当内置异常不足以表达业务错误语义时，创建自定义异常可以让代码更清晰。自定义异常通常继承自 `Exception`，最多加一些初始化参数和描述。

### 最简单的自定义异常

```python
class InsufficientFundsError(Exception):
    """余额不足"""
    pass

def withdraw(balance, amount):
    if amount > balance:
        raise InsufficientFundsError(f"余额 {balance}，无法取出 {amount}")
    return balance - amount
```

使用时与内置异常完全一致：

```python
try:
    balance = withdraw(100, 150)
except InsufficientFundsError as e:
    print(e)
```

### 添加额外属性

为了携带更多上下文信息，可以改写 `__init__`：

```python
class ConfigError(Exception):
    def __init__(self, message, config_key):
        super().__init__(message)
        self.config_key = config_key

# 抛出
raise ConfigError("配置项缺失", "database_url")
```

在 `except` 中就可以根据 `e.config_key` 做特定处理。

### 异常层次的自定义

可以建立一个项目专属的基类异常，然后派生各个具体异常，方便统一捕获。

```python
class MyAppError(Exception):
    """应用根异常"""
    pass

class DatabaseError(MyAppError):
    pass

class ConnectionError(DatabaseError):
    pass
```

这样一来，`except MyAppError` 可以捕获该项目抛出的所有错误，而 `except DatabaseError` 仅捕获数据库相关异常。这种结构在大型库和框架中非常常见。

设计自定义异常时遵守两个原则：

1. **命名以 `Error` 结尾**（符合直觉的惯例）。
2. **提供有意义的消息**，必要时携带诊断数据，便于排查。
